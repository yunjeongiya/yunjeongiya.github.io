어느 날 밤, 팀 채팅방에 내 GitHub 잔디가 올라왔다.

## "어케 하루에 평균 20 commit을 하죠?"

![팀원들의 반응](/assets/images/posts/041-commits-not-productivity/chat-reaction.png)

![팀원들의 반응 2](/assets/images/posts/041-commits-not-productivity/chat-reaction-2.png)

"컨트리뷰션 기계", "GOAT" 같은 반응이 달렸다. 재밌게 봤는데, 문득 궁금해졌다. **진짜 그때가 가장 생산적인 시기였을까?**

직감적으로 "요즘이 오히려 더 잘 되는 것 같은데"라는 느낌이 있었다. 근거 없는 착각일 수도 있으니, 실제 데이터를 까봤다.

---

## 6개월치 git 로그를 분석해봤다

작업일지 77건, git 커밋 로그 전체를 주 단위로 집계했다.

### 커밋 수 추이

| 기간 | 일평균 커밋 | 성격 |
|------|-----------|------|
| 9월 (초기) | 52.0 | 탐색적 개발 |
| **10~11월 (피크)** | **48.5** | 대량 코드 생산 |
| 12~1월 | 산발적 | 학기말/방학 |
| 2월 (재가동) | 28.8 | 기반 정비 |
| **3월 (현재)** | **21.6** | 기능 집중 |

10~11월에는 하루에 50커밋 가까이 찍었다. 3월에는 22커밋도 안 된다. **커밋만 보면 생산성이 반토막 난 것처럼 보인다.**

### 그런데 실제 산출물은?

올해 2월부터 Feature 추적 시스템을 도입해서 "완료된 기능 단위"를 정확히 세고 있다. 3월 기준:

- **일평균 7.0개 Feature 완료** (최대 11개/일)
- Feature당 커밋 수: **~3.1개**

10~11월에는 Feature 단위 추적이 없어서 정확한 비교는 불가능하다. 다만 작업일지를 분석해보면, 하나의 기능에 커밋이 15~20개씩 들어간 흔적이 많았다. 커밋 → 버그 발견 → 수정 커밋 → 또 발견 → 또 수정의 반복.

10~11월은 커밋을 많이 "쳐야만 했던" 시기였고, 3월은 적게 쳐도 같은 결과가 나오는 시기라는 해석이 가능하다.

---

## 좀 더 명확한 지표: post-commit hotfix

커밋 수나 Feature 수보다 더 직관적인 효율 지표가 있다. "커밋 후 즉시 수정한 횟수".

| 기간 | 커밋 후 즉시 수정 | 사례 |
|------|-----------------|------|
| 10~11월 | **일 2~3건** | 커밋 → 7분 후 버그 수정 → 8분 후 또 수정 |
| 3월 | **일 0.3건** | 3일간 총 2건 (사소한 누락) |

10월 30일의 실제 기록:
```
06:18 기능 커밋
06:25 버그 수정 (user.id vs user.data.id 타입 불일치)
06:33 추가 수정 (다이얼로그 중복)
```

15분 안에 같은 기능을 3번 커밋했다. 이게 당시에는 일상이었다.

3월에는 이 패턴이 거의 사라졌다. 물론 여전히 실수는 하지만, 빈도가 확연히 다르다.

---

## 뭐가 달라졌나: 6가지 변화

### 1. 프로세스 자동화

6개월 전에는 프로젝트 규칙을 하나의 파일에 담고, AI에게 매번 수동으로 컨텍스트를 설명했다.

지금은:
- **Feature 추적 시스템**: 208개 Feature가 ID로 관리되고, 작업 범위가 자동으로 명확해진다
- **Scoped Rules**: 백엔드 파일을 열면 Spring Boot 규칙이, 프론트엔드 파일을 열면 React 규칙이 자동 로딩된다
- **Hooks**: 커밋 시 자동 검증, 요구사항 추적 리마인더가 걸려 있다
- **Skills 16개**: `/finish`, `/inspect`, `/test` 같은 반복 워크플로가 한 명령으로 실행된다

이 시스템들을 만드는 데 들인 시간이, 이후의 모든 작업에서 복리로 돌아오는 구조다.

### 2. "Commit-then-Fix" 사이클의 단축

10~11월에는 코드를 작성하고, 커밋하고, 실행해보고, 버그를 발견하고, 수정 커밋을 올리는 사이클이 일상이었다. 테스트를 건너뛰는 일도 잦았다 (`./gradlew build -x test`).

지금은 커밋 전에 `/inspect` (멀티패스 코드 리뷰)와 `/test`로 한 단계를 더 거친다. 완벽하진 않지만, 커밋 후 발견되는 문제가 확실히 줄었다.

### 3. 10~11월의 삽질이 기반이 됐다

이걸 빼놓으면 안 된다. **10~11월에 많이 부딪혔기 때문에 지금이 가능하다.**

10월에 가장 비효율적이었던 날은 리팩토링 후폭풍으로 하루 종일 수습만 한 날이었다. 하나의 구조 변경이 7개 쿼리를 파손시키고 100개 이상의 컴파일 에러를 만들었다. 그 경험이 있어서 2월에 기술 부채를 체계적으로 정리할 수 있었다:

- dead code 5,156줄 삭제
- campus 인증 체계 31개 endpoint 마이그레이션
- ErrorCode 표준화

기반이 깨끗해지니 새 기능을 얹어도 기존 코드가 부서지지 않는다.

### 4. Vertical Slice 체화

예전에는 "오늘 백엔드, 내일 프론트"로 나눠 작업했다. 이건 컨텍스트 스위칭 비용이 크고, 다음 날 이어붙일 때 통합 버그가 터진다.

지금은 모든 Feature가 **서버 + 프론트엔드 + 문서를 한 단위로** 완성된다. 이것도 처음부터 이렇게 한 건 아니고, 절반만 된 채로 다음 날 이어붙이다가 버그가 터지는 경험을 몇 번 하고 나서 자연스럽게 바뀌었다.

### 5. 도메인 지식 축적

208개 Feature를 거치면서 코드베이스 전체의 구조가 머릿속에 들어왔다. 새 기능을 어디에 넣어야 하는지, 어떤 패턴이 문제를 일으키는지(LazyInitException, N+1, timezone 등) 설계 단계에서 감이 온다.

이건 시스템이 아니라 순수하게 시간과 경험의 결과다. 지름길은 없었다.

### 6. Feature 체이닝

3월의 작업을 보면, 하루 안에서 Feature들이 자연스럽게 연결된다:

```
알림톡 모달(F185) → 아키텍처 문제 발견 → 리팩토링(F186)
대기 시스템(F192) → 탭 UI(F197) → 커스텀 모달(F192 Phase 2)
환불 계산(F200) → 차액 청구서 자동 생성(F202)
```

앞 Feature의 컨텍스트가 그대로 다음에 이어지니 워밍업 시간이 거의 없다. 이건 의식적으로 설계한 것도 있지만, 프로젝트가 성숙하면서 기능 간 연결고리가 자연스럽게 늘어난 측면도 있다.

## 숫자 요약

| 지표 | 10~11월 | 3월 | 변화 |
|------|---------|-----|------|
| 일평균 커밋 | 48.5 | 21.6 | -55% |
| 일평균 Feature 완료 | (추적 없음) | 7.0 | - |
| Feature당 커밋 | ~15~20 (추정) | ~3.1 | -80% (추정) |
| post-commit hotfix | 일 2~3건 | 일 0.3건 | -87% |
| 심야 작업 | 빈번 (새벽 4시) | 없음 | 소멸 |
| 주간 작업일 | 2~6일 (불규칙) | 5일 (안정) | 안정화 |

10~11월 데이터에는 Feature 추적이 없어서 일부 수치는 추정치다. 정확한 동일 조건 비교는 아니라는 점은 감안해야 한다.

## 배운 것

GitHub 잔디가 진하면 기분은 좋다. 그리고 커밋이 많다는 건 분명히 열심히 했다는 뜻이다.

하지만 커밋이 많다는 건 "많이 고쳤다"는 뜻일 수도 있다. 한 번에 맞추면 커밋은 적어진다.

내 6개월 데이터에서 보이는 패턴은 이거다:

> **10~11월의 대량 커밋은 "노력의 시기"였고, 3월의 적은 커밋은 그 노력 위에 시스템이 얹어진 결과다.**

커밋이 줄었지만 산출물은 늘었다. 새벽 4시까지 안 해도 된다. 리팩토링 후폭풍에 하루를 날리지 않는다.

차이를 만든 건 크게 두 가지였다:
1. **시스템**: Feature 추적, 자동 코드 리뷰, scoped rules, hooks 등이 쌓이면서 반복 비용이 줄었다
2. **경험**: 10~11월에 부딪힌 모든 문제가 지금의 판단력이 됐다

둘 다 시간이 필요했다. 10~11월의 삽질 없이 3월의 효율은 없었을 거다.

잔디 색깔보다 중요한 건 **"한 커밋당 얼마나 많은 가치를 전달하고 있는가"**일 것이다. 이 숫자가 시간이 갈수록 커지고 있다면, 괜찮은 방향으로 가고 있는 거 아닐까.
