어느 날 밤, 팀 채팅방이 시끄러워졌다.

## "어케 하루에 평균 20 commit을 하죠?"

![팀원들의 반응](/assets/images/posts/041-commits-not-productivity/chat-reaction.png)

![팀원들의 반응 2](/assets/images/posts/041-commits-not-productivity/chat-reaction-2.png)

누군가 내 GitHub 잔디를 캡쳐해서 올렸다. "컨트리뷰션 기계", "GOAT", "8배 차이" 같은 반응이 달렸다. 기분이 나쁠 리 없었지만, 하나 재미있는 사실이 있다.

**팀원들이 놀란 그 시점이, 사실 내 커밋이 가장 적은 시기였다.**

---

## 궁금해서 데이터를 까봤다

체감적으로 "요즘 일이 잘 된다"는 느낌이 있었다. 근거 없는 자기만족일 수도 있으니, 실제 git 로그와 작업일지를 분석해봤다. 6개월치, 주 단위로.

### 커밋 수 추이

| 기간 | 일평균 커밋 | 성격 |
|------|-----------|------|
| 9월 (초기) | 52.0 | 탐색적 개발 |
| **10~11월 (피크)** | **48.5** | 대량 코드 생산 |
| 12~1월 | 산발적 | 학기말/방학 |
| 2월 (재가동) | 28.8 | 기반 정비 |
| **3월 (현재)** | **21.6** | 기능 집중 |

10~11월에는 하루에 50커밋 가까이 찍었다. 3월에는 22커밋도 안 된다. **커밋만 보면 생산성이 반토막 난 것처럼 보인다.**

### 그런데 실제 산출물은?

3월부터 Feature 추적 시스템을 통해 "완료된 기능 단위"를 정확히 세고 있다. 결과:

- **일평균 7.0개 Feature 완료** (최대 11개/일)
- Feature당 커밋 수: **~3.1개**

10~11월에는 Feature 단위 추적이 없었지만, 작업일지를 분석해보면 하나의 기능에 커밋이 15~20개씩 들어갔다. 커밋 → 버그 발견 → 수정 커밋 → 또 발견 → 또 수정의 반복이었다.

**즉, 10~11월은 커밋을 많이 "쳐야만 했던" 시기이고, 3월은 적게 쳐도 되는 시기다.**

---

## 결정적 지표: post-commit hotfix

가장 직관적인 효율 지표는 "커밋 후 즉시 수정한 횟수"다.

| 기간 | 커밋 후 즉시 수정 | 사례 |
|------|-----------------|------|
| 10~11월 | **일 2~3건** | 커밋 → 7분 후 버그 수정 → 8분 후 또 수정 |
| 3월 | **일 0.3건** | 3일간 총 2건 (사소한 누락) |

10월 30일의 실제 기록:
```
06:18 기능 커밋
06:25 버그 수정 (user.id vs user.data.id 타입 불일치)
06:33 추가 수정 (다이얼로그 중복)
```

3월에는 이런 패턴이 거의 없다. **한 번에 맞춘다.**

---

## 왜 이렇게 됐나: 6가지 원인

### 1. 프로세스 자동화

6개월 전에는 CLAUDE.md 파일 하나에 모든 규칙을 담고, AI에게 매번 수동으로 컨텍스트를 설명했다.

지금은:
- **Feature 추적 시스템**: 208개 Feature가 ID로 관리되고, 작업 범위가 명확하다
- **Scoped Rules**: 백엔드 파일을 열면 Spring Boot 규칙이, 프론트엔드 파일을 열면 React 규칙이 자동 로딩된다
- **Hooks**: 커밋 시 자동 검증, 요구사항 추적 리마인더가 걸려 있다
- **Skills 16개**: `/finish`, `/inspect`, `/test` 같은 반복 워크플로가 한 명령으로 실행된다

관리 오버헤드가 시스템에 흡수되면서, 나는 코드에만 집중하면 된다.

### 2. "Commit-then-Fix" 패턴의 소멸

10~11월에는 코드를 작성하고, 커밋하고, 실행해보고, 버그를 발견하고, 수정 커밋을 올리는 사이클이 일상이었다. 테스트를 건너뛰는 일도 잦았다 (`./gradlew build -x test`).

지금은 커밋 전에 `/inspect` (멀티패스 코드 리뷰)와 `/test`가 문제를 잡아준다. **버그가 커밋에 도달하기 전에 제거된다.**

### 3. 기술 부채 청산

10월에 가장 비효율적이었던 날은 리팩토링 후폭풍으로 **하루 종일 수습만 한 날**이었다. 하나의 구조 변경이 7개 쿼리를 파손시키고 100개 이상의 컴파일 에러를 만들었다.

2월에 이 부채를 집중적으로 정리했다:
- dead code 5,156줄 삭제
- campus 인증 체계 31개 endpoint 마이그레이션
- ErrorCode 표준화

**기반이 깨끗하니 새 기능을 얹어도 부서지지 않는다.**

### 4. Vertical Slice 체화

예전에는 "오늘 백엔드, 내일 프론트"로 나눠 작업했다. 이건 컨텍스트 스위칭 비용이 크고, 다음 날 이어붙일 때 통합 버그가 터진다.

지금은 모든 Feature가 **서버 + 프론트엔드 + 문서를 한 단위로** 완성된다. 어떤 기능도 "절반만 된 상태"로 하루를 넘기지 않는다.

### 5. 도메인 지식 축적

208개 Feature를 거치면서 코드베이스의 멘탈 모델이 완성됐다. 새 기능의 위치를 즉시 파악하고, 흔한 에러 패턴(LazyInitException, N+1, timezone 문제)을 설계 단계에서 회피한다.

10~11월에 하루를 소비한 디버깅이, 3월에는 발생 자체가 안 된다.

### 6. Feature 체이닝

3월의 작업을 보면, 하루 안에서 Feature들이 의도적으로 연결된다:

```
알림톡 모달(F185) → 아키텍처 문제 발견 → 리팩토링(F186)
대기 시스템(F192) → 탭 UI(F197) → 커스텀 모달(F192 Phase 2)
환불 계산(F200) → 차액 청구서 자동 생성(F202)
```

앞 Feature의 컨텍스트가 그대로 다음 Feature에 이어지니, **매번 "어디까지 했더라"부터 시작하지 않는다.**

## 결론: 커밋 수는 생산성이 아니다

GitHub 잔디가 진하면 기분은 좋다. 팀원들의 "GOAT" 반응도 나쁘지 않다.

하지만 **커밋이 많다는 건 "많이 고쳤다"는 뜻일 수도 있다.** 한 번에 맞추면 커밋은 적어진다.

내 6개월 데이터가 보여주는 건 이거다:

> **10~11월은 "열심히 한 시기"였고, 3월은 "잘 한 시기"다.**

커밋 55%가 줄었지만 산출물은 늘었다. 새벽 4시까지 안 해도 된다. 리팩토링 후폭풍에 하루를 날리지 않는다.

차이를 만든 건 개인의 노력이 아니라 **시스템**이었다. Feature 추적, 자동 코드 리뷰, scoped rules, hooks, 기술 부채 정리 — 이런 것들이 쌓이면서 같은 노력이 더 큰 결과를 만드는 구조가 됐다.

잔디 색깔보다 중요한 질문은 이거다: **"한 커밋당 얼마나 많은 가치를 전달하고 있는가?"**

그 답이 시간이 갈수록 커지고 있다면, 올바른 방향으로 가고 있는 거다.
