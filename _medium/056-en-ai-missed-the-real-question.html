The Bug AI Fixed 4 Times But Never Solved — Until a Human Asked One Question

This post is also available on my blog. (Korean version here)

AI Coding Claude Code Architecture API Design Meta-cognition

I was building a feature with an AI coding assistant. We went through four rounds of code review, and each round surfaced a new edge case. Every time, the AI said "fixed it." Every time, a different variant of the same class of bug popped up in the next round.

Right before kicking off the fifth review, I asked one question:

"Can't we just make this problem impossible in the first place?"

That single question changed the entire approach. All the patching code accumulated over four rounds became unnecessary.

The Setup

I run a learning management app for a tutoring center. Students log their study hours, and the app displays them on a 10-minute heatmap grid:

┌──────┬────┬────┬────┬────┬────┬────┐
│      │ :00│ :10│ :20│ :30│ :40│ :50│
├──────┼────┼────┼────┼────┼────┼────┤
│ 3 PM │████│████│████│████│    │    │  ← English (blue)
│ 4 PM │▓▓▓▓│▓▓▓▓│▓▓▓▓│    │    │    │  ← Math (yellow)
└──────┴────┴────┴────┴────┴────┴────┘
         ↑ student selects this range to reassign the subject

The feature: let students select a time range on the heatmap and change which subject it's assigned to.

The core problem: the UI displays in 10-minute slots, but the server stores exact timestamps down to the second.

Heatmap (UI)Server Data
Start3:00:00 PM3:02:15 PM
End4:10:00 PM4:07:30 PM
Precision10 minseconds

When the frontend picks a slot, it has to convert it back to an exact timestamp to send to the server. That's where the trouble started.

What the AI Built

The AI designed this data flow:

Frontend: exact time → 10-min slot (for display) → back to exact time → send to server

This is JPEG compression. Converting 3:02:15 PM to a 10-minute slot gives you 3:00. Converting back gives 3:00:00. The original 2 minutes and 15 seconds are gone forever.

Four bugs cascaded from this gap.

Four Patches, Same Class of Bug

Round 1: UTC Offset

When converting slots to timestamps, KST (+9) was calculated manually. Crossing midnight shifted the date by a day.

AI's fix: import a timezone library for conversion.

Round 2: String Format

The server returned 15:02:15, but the frontend compared it against 15:02:15.000. Mismatch. Matching failed.

AI's fix: add a normalize function before comparison.

Round 3: Boundary Validation

The outer boundaries of a 10-minute slot fell outside the actual data range. Slot 15:00-15:10 didn't cover actual data starting at 15:02:15. Server rejected.

AI's fix: add clamping logic — if out of range, snap to the actual data boundary.

Round 4: Server Validation Failure

The clamped timestamps didn't match the server's precise expectations. The server knows the exact original time, and the "corrected" value from the frontend was awkwardly off.

AI's fix: more sophisticated clamping + edge case handling.

After the fourth fix, the AI summarized:

"The slot index ↔ actual time conversion layer is inherently lossy. Clamping has closed the gap."

It worked. But the code was now layered with correction functions: slotToUtcTime(), normalizeTimestamp(), clampToWorkLogBoundary().

One Question

I asked:

"Can't we just make it not lossy in the first place?"

The AI paused, rethought from scratch, and came back with:

"If what the frontend sends isn't a reverse-converted slot time, but the original record's ID plus split points, the reverse conversion simply doesn't exist. No lossy conversion, no lossy bugs."
Before: frontend calculates timestamps and sends them (lossy)
  → slotToUtcTime(69) → "15:30:00" → send to server → server rejects

After: frontend sends intent only (lossless)
  → { workLogId: 42, splitPoints: ["15:30"] } → server builds from originals

The frontend just says "split here." The server uses its own exact timestamps to do the rest. slotToUtcTime, normalizeTimestamp, clampToWorkLogBoundary — all deleted.

Why the AI Never Asked This Question

Across four review rounds, here's what the AI did:

  1. Accurately diagnosed the bug
  2. Accurately fixed that bug
  3. Reported "it's resolved now"
  4. When the next bug appeared, went back to step 1

Every step was correct. But stepping up one level to ask "why does this layer need to exist?" never happened.

This isn't so much an AI limitation as it is a pattern trap. When code review says "this function has a bug," the natural response is "let's fix that function" — not "should this function exist?" Humans fall into the same trap. The difference is that humans have frustration as a signal. The feeling of "again?!" leads to "is something fundamentally wrong here?"

AI doesn't get frustrated. It approaches the tenth patch with the same attitude as the first.

Principles From This Experience

1. When Patches Repeat, Question the Layer

If you're adding complexity to bridge a gap, first check whether that gap needs to exist.
SymptomQuestion to Ask
Validation fixes keep recurringShould this data go through this validation at all?
Format conversion bugs keep recurringShould this conversion exist?
Null checks keep recurringShould this null be possible?

"Change your hypothesis" is a good rule, but you can circle at the same level. What you really need is to question the architecture.

2. Clients Send Intent, Servers Own the Data

This pattern already has names in software architecture:

  • Task-Based UI: the client sends work intent, not mutated state
  • Intent-Based API: POST /resource/action {intent} instead of PUT /resource {fields}
  • CQRS Command: a Command object expressing user intent; the server decides state transitions

Core principle: the side that owns the authoritative data should perform the computation. The moment the client guesses and reconstructs the server's data, lossy conversion begins.

3. When Working With AI, Humans Own the Meta-cognition

AI excels at "this code has a bug → fix it." But it rarely asks "should this code exist?" That's the difference between faithfully following instructions and questioning the instructions themselves.

When collaborating with AI coding assistants, the role of stepping back to see the whole picture is still the human's job. When the AI proposes a third patch, interrupting with "wait, why are we fixing this for the third time?" — that's the most valuable contribution you can make.

After this experience, I added a rule to my AI coding assistant's configuration:

After 2 failed fixes in the same area, review the architecture: If you fail twice in the same area, question the architecture, not the hypothesis. "Should this conversion/layer/validation exist at all?"

I also created a hook that auto-reminds "should this layer exist?" when debugging keywords are detected. Think of it as simulating "frustration" for the AI.
